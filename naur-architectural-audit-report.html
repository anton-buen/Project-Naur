1 — Critical Protocol & Schema Inconsistencies
Mismatched dictionary keys, Pydantic-schema blind spots, and transactional vulnerabilities that can produce silent data corruption or tool-invocation failures.
Critical
Array join produces literal \n stored in SQLite, not real newlines
mcp_server.py : lines 101–104 → state_manager.py : update_constraint → app.py : parse_markdown()

The MCP schema declares constraint_text and business_impact as JSON arrays of strings. The dispatcher joins them with a Python newline character ("\n"): 

text = "\n".join([f"- {t}" for t in t_data])

When the MCP client (IBM Bob) serializes the tool arguments over the stdio transport as a JSON payload, the resulting string that arrives in arguments is already a decoded Python string — so the join itself is correct. However, if the LLM agent chooses to pass a pre-serialized JSON string instead of a native array (a common failure mode with function-calling models), the isinstance(t_data, list) guard silently falls through to str(t_data), which produces the raw Python repr "['bullet one', 'bullet two']" — stored verbatim in SQLite and rendered as broken text in the Streamlit card.

Additionally, even on the happy path, the joined string stored in constraint_text begins with - bullet. When parse_markdown() in app.py processes this, it detects the leading - and wraps it in a <li> tag — which is correct. But the MCP layer has already added the - prefix, meaning every consumer downstream will always receive pre-formatted markdown it cannot strip. If any other consumer reads the raw DB value expecting plain text, it will receive markdown syntax.

Recommended Fix
Store arrays as raw joined strings without the - prefix at the MCP layer; let parse_markdown() be the sole formatting authority. Change the join in mcp_server.py to: 
text = "\n".join(t_data) if isinstance(t_data, list) else str(t_data)
Add a validation guard to reject non-list payloads with a structured error instead of silently coercing via str().
Medium
Column name aliasing gap: DB schema vs. Python API surface mismatch
state_manager.py : get_constraints() lines 117–124 vs. app.py : lines 332–334, 372–374

The SQLite table domain_constraints stores columns named business_impact and risk_level. The get_constraints() function re-aliases them on return: 

"business": r["business_impact"],
"risk":     r["risk_level"],

Meanwhile, the MCP tool schema names its parameters business_impact and risk_level, and the update_constraint() function signature uses a positional argument named risk — not risk_level. This creates three distinct names for the same concept across the call chain: 

MCP schema key    → "risk_level"       (Tool inputSchema)
state_manager arg → risk               (update_constraint parameter)
DB column         → risk_level         (domain_constraints table)
Python dict key   → "risk"             (get_constraints return value)
app.py access     → data.get("risk")   (presentation layer)

This is not a runtime bug today because the aliasing is internally consistent. However, it is a maintenance trap: any future developer adding a new read path or a second consumer of get_constraints() will face a non-obvious key name that does not match the DB column or the tool schema, increasing the probability of a KeyError / silent None regression. The MCP tool and the DB column agree on business_impact/risk_level; the Python API should surface the same names.

Recommended Fix
Rename the aliased keys in get_constraints() to match the DB column names, and update app.py accordingly: 
# state_manager.py
"business_impact": r["business_impact"],
"risk_level":      r["risk_level"],

# app.py
data.get("business_impact", "")
data.get("risk_level", "LOW")
Also rename the update_constraint() parameter from risk to risk_level for full chain consistency.
Medium
clear_ledger() issues three DELETEs inside one with block but without explicit transaction demarcation
state_manager.py : clear_ledger() lines 175–178

Python's sqlite3 module uses implicit transaction management. The with conn: context manager commits on exit and rolls back on exception — but only for DML statements that have been issued since the last implicit commit. SQLite's default isolation level means the three DELETE statements may be auto-committed individually if Python's implicit transaction logic issues an implicit BEGIN before each one.

If the second or third DELETE fails (e.g., due to a concurrent reader holding a shared lock beyond the 15-second timeout), the connection context manager will roll back only the in-flight statement, potentially leaving chat_ledger empty while domain_constraints and project_glossary retain stale data. The resulting partial-clear state is invisible to the UI — it will rerun, appear cleared, then immediately re-render stale constraint cards from the un-cleared tables.

Recommended Fix
Wrap all three DELETEs in an explicit BEGIN/COMMIT to guarantee atomicity: 
with get_connection() as conn:
    conn.execute("BEGIN")
    conn.execute("DELETE FROM chat_ledger")
    conn.execute("DELETE FROM domain_constraints")
    conn.execute("DELETE FROM project_glossary")
    conn.execute("COMMIT")
Or equivalently, use isolation_level=None on the connection and manage the transaction manually.
2 — Dead Code & Architectural Drift
Legacy logic, unused imports, and boundary violations that erode the decoupled separation of concerns.
Low
init_db() executes at module import time — untestable side-effect
state_manager.py : line 184

The bare call init_db() at the bottom of state_manager.py runs the moment anything imports the module, including unit test runners, linters, or the MCP server subprocess. This creates a physical side-effect (creates/opens naur_state.db on disk) purely as a consequence of an import statement.

The architectural contract of the system states that state_manager is the exclusive gatekeeper. But a gatekeeper that opens the gate unconditionally on construction is not a gatekeeper — it is an auto-initializer. Any test that imports state_manager will produce a stale or polluted naur_state.db in the project root.

Recommended Fix
Move init_db() to the application entrypoints that own startup: the top of app.py and the main() function in mcp_server.py. Remove the bare call from the module body. 
# app.py — top of file, after imports
sm.init_db()

# mcp_server.py — inside main()
async def main() -> None:
    sm.init_db()
    async with stdio_server() as (r, w): ...
Low
Chat avatar role-detection reads the wrong variable — always falls through to "U"
app.py : lines 413–417

The avatar initial detection logic checks for role tags in clean_content: 

if "[Product" in clean_content:  avatar_initials = "PM"
elif "[Frontend" in clean_content: avatar_initials = "FE"
...

But clean_content has already been processed by two re.sub() calls that strip [Context: ...] and [Governance: ...] tags. The role tag ([Product Manager], [Frontend Engineer], etc.) is prepended to the message in the chat input handler at line 432 as f"[{active_role}] {user_intent}" and is not stripped by either regex — so clean_content does still contain the role tag. This means the detection does work today.

However, the strip regexes use a leading space: ' \[Context: .*?\]'. A message that starts with [Context: (no leading space) would not be stripped, while a role tag at position 0 is never preceded by a space. The logic is accidentally correct but fragile — a future change to either the stamping format or the strip pattern will silently break avatar assignment, and there is no test coverage to catch it.

Recommended Fix
Extract the role from the tag directly rather than doing substring detection on the full message string: 
role_match = re.match(r'^\[([^\]]+)\]', msg["content"])
role_tag = role_match.group(1) if role_match else ""
avatar_map = {"Product Manager": "PM", "Frontend Engineer": "FE",
              "Backend Engineer": "BE", "Data Scientist": "DS", "UI/UX Designer": "UI"}
avatar_initials = avatar_map.get(role_tag, "U")
Low
Dead dependency surface: pandas, pyarrow, narwhals, altair are Streamlit transitive deps with no direct usage
requirements.txt

requirements.txt pins pandas==3.0.3, pyarrow==24.0.0, narwhals==2.23.0, and altair==6.2.2 as top-level dependencies. None of these are directly imported in app.py, state_manager.py, or mcp_server.py. They are pulled in transitively by Streamlit.

Pinning transitive dependencies as first-class entries inflates the attack surface and causes unnecessary dependency conflict noise on upgrades. More critically, pyarrow==24.0.0 is a non-existent version at time of writing (latest stable is 18.x) — this will cause pip install -r requirements.txt to fail outright in a clean environment.

Recommended Fix
Remove altair, narwhals, pyarrow, and pandas from requirements.txt and let Streamlit resolve its own transitive graph. The direct dependency set for this application is: streamlit, mcp, and optionally python-dotenv.
3 — Resilience & Sanitization Fixes
String parsing fragility, empty-state rendering gaps, and presentation layer hardening recommendations.
Critical
Glossary .replace() chain fires on raw HTML output of parse_markdown(), not on source text
app.py : line 400

The glossary rendering pipeline applies string replacements after Markdown-to-HTML conversion: 

clean_def = parse_markdown(definition).replace("In Project Naur, ", "").replace("Project Naur", "This project")

This is architecturally inverted. parse_markdown() returns an HTML string with inline styles, <div>, <b>, etc. The .replace() calls then perform a naive substring search on that HTML. This means:

1. If the text "Project Naur" appears split across an HTML tag boundary (e.g., Project <b>Naur</b>), neither replace will fire, leaving the raw text in place — silent failure. 2. If definition is an empty string or None, parse_markdown() returns "", and the .replace() calls on an empty string are harmless but semantically undefined. 3. If definition is None (the DB returned a NULL), parse_markdown()'s if not text guard returns "" safely — but only because of that guard. Any future refactor that reorders those two operations will crash with AttributeError: 'NoneType' object has no attribute 'replace'.

Recommended Fix
Apply text sanitization before Markdown conversion, on the raw source string: 
raw_def = (definition or "").replace("In Project Naur, ", "").replace("Project Naur", "This project")
clean_def = parse_markdown(raw_def)
Medium
parse_markdown() is not safe against inputs containing unbalanced HTML from the DB
app.py : parse_markdown() — called with raw DB values from deep_dive, constraint_text, business_impact

parse_markdown() renders its output via st.markdown(..., unsafe_allow_html=True). The deep_dive field in particular is defined in the MCP schema as a freeform "string" with the description "Put long-form technical explanations here." — there is no length cap, no HTML stripping, and no escaping applied before the value is stored in SQLite or rendered.

If an LLM agent writes raw HTML into deep_dive (e.g., <script> or a malformed <div>), it will be stored verbatim and injected directly into the Streamlit page via unsafe_allow_html=True. While Streamlit's server-side environment limits the impact of XSS, a malformed tag can fully break the card layout in the browser.

Recommended Fix
Apply a lightweight HTML escape on values read from the DB before passing to parse_markdown(), since parse_markdown() already re-generates its own safe HTML tags: 
import html
tech_text = parse_markdown(html.escape(data.get("text", "") or ""))
deep_dive = parse_markdown(html.escape(data.get("deep_dive", "") or ""))
The or "" coercion also eliminates the None propagation risk if a DB row has a NULL column.
Medium
Empty-state: highest_risk computation iterates a potentially heterogeneous list without full type guarding
app.py : lines 307–313

The risk aggregation block builds all_risks from constraints.values() plus the optional global_summary, then filters with isinstance(v, dict). This guard is correct but the code reaches this block only when if constraints or global_summary is truthy — meaning at least one of them is non-empty. If constraints is empty ({}) and only global_summary is present, all_risks will contain only global_summary.

The real risk is that global_summary.get("risk") returns None if the DB row's risk_level column is NULL (which is possible since INSERT OR REPLACE does not enforce NOT NULL). The max(..., default="LOW") catches the empty-iterator case but not the None-from-.get() case: risk_priority.get(None, 0) returns 0, so None wins a tie as "LOW"-equivalent — silently producing a None value passed into the f-string at line 319, which renders as the literal text "Alignment Risk: None" in the badge.

Recommended Fix
Coerce the risk value to a known default at the read site inside get_constraints(): 
"risk": r["risk_level"] or "LOW",
And identically for the key rename fix above: "risk_level": r["risk_level"] or "LOW". This keeps the fallback logic in the data layer rather than scattering it across the presentation layer.
Low
External avatar URL dependency — offline / air-gapped environment failure
app.py : lines 419, 424

Both the human and assistant chat avatars are loaded from ui-avatars.com at render time. This is a runtime HTTP dependency inside a tool advertised as a local, decoupled, air-gap-safe architectural linter. In environments without internet access (common in enterprise settings where this tool is most likely deployed), every chat message will render with a broken image icon instead of the intended avatar, degrading the visual hierarchy of the conversation thread.

Recommended Fix
Generate initials avatars as inline SVG data URIs — no network call required: 
def make_avatar_svg(initials: str, bg: str, fg: str) -> str:
    svg = (f'<svg xmlns="http://www.w3.org/2000/svg" width="40" height="40">'
           f'<circle cx="20" cy="20" r="20" fill="{bg}"/>'
           f'<text x="20" y="25" text-anchor="middle" '
           f'font-size="14" font-weight="bold" fill="{fg}">{initials}</text></svg>')
    import base64
    return "data:image/svg+xml;base64," + base64.b64encode(svg.encode()).decode()
